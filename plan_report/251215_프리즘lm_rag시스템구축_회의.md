🧑‍💼 시니어 개발자(사회) 통합 회의록 (Markdown)

> 범위: 지금까지 대화 전체 + 전문가 전원 참여 의견 + 최종 MUST/SHOULD 반영본




---

0) 서비스 한 줄 정의

강의/자료 지식베이스를 기반으로 사용자가 작성한 글을 루브릭(평가기준) 기준으로 근거 인용하며 평가·진단하고, 부족한 점과 개선 행동을 제공하는 서비스

단순 “문서 업로드 → 검색 → 보여주기”가 아니라, 기준 준수 여부를 점검하는 평가 서비스



---

1) 회의 참여자(전문가) 및 역할

1.1 핵심 참여자(필수)

도메인 전문가(글쓰기 강의자/편집장급)

역할: 강의 내용을 **루브릭(평가기준표)**로 공식화, 예시/반례/최소통과선 정의


LLM/RAG 아키텍트(시니어)

역할: 전체 아키텍처 설계(인덱싱/리트리벌/리랭킹/프롬프트/가드레일), 비용·품질 트레이드오프 결정


정보검색(IR)·검색 엔지니어

역할: 청킹/메타데이터/하이브리드 검색/리랭커 설계로 근거 회수율(Recall) 보장


평가(Evaluation) 전문가

역할: 품질 지표 정의, 골드셋 구축, “점수의 신뢰성/재현성” 검증 체계 설계


데이터 엔지니어

역할: 강의자료 수집→정제→버전관리→인덱싱 파이프라인, 로그/감사 추적(재현성) 구축


보안/개인정보(Privacy/Security)

역할: ACL/테넌시, PII 마스킹, 저장/보관 정책, 접근 통제 설계


PM(제품 매니저)/기획

역할: MVP 범위 확정, UX 산출물 정의(결과 화면/지표), 출시/운영 기준 수립



1.2 옵션 참여자(권장)

UX 라이터/콘텐츠 디자이너: 피드백 문구(코칭/교정/채점) 설계, 사용자 이탈 방지

프론트엔드 리드: 근거 하이라이트/코멘트 UI, 버전 비교 UI

MLOps/SRE: 배포/모니터링/비용 최적화/캐시/장애 대응

법무/컴플라이언스: 강의자료 저작권/이용범위, 책임 고지

QA/테스트: 루브릭별 회귀 테스트 자동화



---

2) 핵심 합의(원칙)

근거 우선: 모든 평가/지적/점수는 반드시 강의 근거 인용으로 연결

근거 없으면 판단 금지: 근거 부족 시 **‘근거 부족’**으로만 출력

재현 가능: 어떤 자료/버전/근거로 평가했는지 추적 가능

RAG만으로는 부족: “검색(Retrieval)” + “채점(Judge)”의 2단 파이프라인이 정석



---

3) MUST 구현(필수 요구사항)

3.1 강의 자료(지식베이스) 입력 파이프라인

업로드/수집(PDF/문서/노션/웹 등)

정제/품질관리: 중복 제거(hash), 최신본 우선, 깨짐/오탈자 처리

청킹(Chunking): 헤딩/슬라이드/섹션 보존 + 길이(토큰) 기준 + overlap(필요 시)

메타데이터:
doc_id, title, section, tags, version, updated_at, source, access_level

임베딩 + 인덱싱: text + embedding(float[]) + metadata

버전 관리: 문서/청크/인덱스 버전 추적


3.2 접근 통제(ACL) + 테넌시(Namespace 격리)

ACL: 계약된 사용자만 해당 강사/코스 자료 접근

Namespace 격리: 강사/코스별 지식베이스 논리 분리

Retrieval 단계 필터: tenant_id / access_level / version 필터를 검색 단계에서 적용


3.3 루브릭(평가기준표) 시스템

강의 내용 기반 루브릭 10~20개를 고정 기준으로 관리

항목 구성: 항목명 / 설명 / 통과기준 / 좋은 예 / 나쁜 예(가능하면)

루브릭 버전 관리(v1, v2…)


3.4 평가 실행: RAG + Judge(채점)

1. 사용자 글 입력


2. 루브릭별 평가 쿼리 생성


3. 근거 검색(Retrieval): 항목별 Top-K 근거 회수

하이브리드 검색(BM25+벡터) 또는 리랭커 중 최소 1개는 필수



4. Judge 채점: 루브릭별로

status/score + 근거 인용 + 부족점 + 개선안 생성



5. 실패 모드: 검색 실패/근거 부족 시 안내 + 로그 기록



3.5 결과 출력(고정 JSON 포맷)

최소 필드:

rubric_item

status(통과/보완/미충족) 또는 score

evidence_quotes (+ chunk_id)

gap

recommendations

confidence(선택)


근거 인용 없는 평가/점수는 무효


3.6 UI/UX 최소 필수

사용자 글에 코멘트/하이라이트로 피드백 연결

결과는 상태표(통과/보완/미충족) 우선 표시

근거는 클릭 시 원문(청크)로 연결



---

4) MUST 운영(필수 운영/품질 체계)

평가 지표(Eval): 근거 적합성 / 인용 정확성 / 일관성 / 유용성(최소 2~4개)

로그/감사 추적:

강의/루브릭 버전, Top-K chunk_id, 출력 JSON, 실패율 저장


비용 가드레일/캐시: 임베딩/검색 캐시, 재실행 쿼터

보안/개인정보: PII 마스킹 옵션, 보관 정책(최소 보관), 저작권 보호



---

5) SHOULD 구현(권장: v1~v2)

5.1 평가 모드(프롬프트 모듈화 / Persona)

사용자 선택 3종 권장:

코칭형 / 직설 교정형 / 채점표형


모드에 따라 시스템 프롬프트 모듈 교체


5.2 교육학 메타데이터 고도화(Pedagogy Tagging)

청킹 시 자동 태깅(반자동 가능):

Category: 논리/문법/표현

Difficulty: 상/중/하



5.3 자동 재작성(옵션)

MVP는 “평가+개선안”

재작성은 사용자 선택 시에만



---

6) 전문가별 제안 아이디어(핵심만)

6.1 도메인 전문가

강의 내용을 루브릭화(10~20) + 좋은/나쁜 예 + 최소통과선

점수보다 먼저 피드백 타입(코칭/교정/레드라인) 확정


6.2 LLM/RAG 아키텍트

2단 구조 고정: Retrieval → Judge

출력은 JSON 고정 + “근거 없으면 근거 부족” 가드레일

프롬프트 계약(Contract) 버전 관리


6.3 IR/검색 엔지니어

헤딩/슬라이드 단위 청킹 + 메타데이터 강화

벡터 단독 지양: 하이브리드(BM25+벡터) 또는 리랭커 필수

루브릭별 근거 Top-K를 따로 회수(한 번에 다 찾지 말기)


6.4 평가(Eval) 전문가

지표 4종 권장: 근거적합성/인용정확성/일관성/유용성

골드셋 구축(전문가 정답 피드백)

“근거 인용 없는 점수는 무효” 규칙


6.5 데이터 엔지니어

수집→정제→청킹→임베딩→인덱싱 재현 가능한 파이프라인

doc_version, hash, chunk_id로 추적성 확보

평가 실행 로그를 품질 개선에 재사용


6.6 보안/개인정보

ACL + Namespace 격리 + Retrieval 필터링

사용자 글 PII 마스킹 옵션

저장/보관 정책 명확화(기본 최소 보관)


6.7 PM/기획

MVP 고정: 루브릭 10개 + 근거 인용 + 개선안 3개

우선순위: 신뢰도(근거) > UX > 재작성(옵션)

과금 단위: 평가 횟수/문서 용량/고급 루브릭/팀 기능


6.8 UX 라이터(옵션)

피드백 톤 3단(코칭/교정/채점) 템플릿

지적보다 “다음 행동” 중심 문장 설계


6.9 프론트엔드 리드(옵션)

주석형 코멘트 UI(근거→지적→수정 예시)

버전 비교(수정 전/후 하이라이트)



---

7) MVP 정의(최종 합의)

강의자료 업로드 → 루브릭 10개 → 루브릭별 Top-K 근거 회수 → 근거 인용 평가 → 부족점/개선안 최소 3개 → 실행 로그 저장

ACL/Namespace + “근거 없으면 근거 부족” 규칙 적용



---

8) 다음 회의 확정 안건(5)

1. 루브릭 10~20개 초안 확정


2. 청킹 규칙/메타데이터 스키마 확정(+ 선택: Category/Difficulty)


3. Retrieval 전략(하이브리드 vs 리랭커) 확정


4. 결과 JSON 스키마 + 프롬프트 계약 확정


5. MVP 화면 1장(하이라이트/상태표/근거 링크) 확정
