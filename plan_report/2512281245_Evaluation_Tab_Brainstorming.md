## 2. 현재 평가 탭 분석

### 기능 현황

- **전체 평가**: RAG 기반 문서 전체 평가
- **히스토리**: 과거 평가 기록 조회 및 삭제
- **피드백**: 점수, 판정(Judgment), 업그레이드 플랜
- **액션**: 수정 패치 적용, 재생성 요청

### 현재 문제점

- **UX**: 결과가 텍스트 위주로 무겁고 압도적
- **성능**: 긴 문서 평가 시 진행 상황 피드백 부족
- **비주얼**: 프리미엄 느낌 부족 (현재: 기본 Tailwind UI)

## 3. 디렉터 비전 및 전문가 피드백

### 🎯 디렉터 요청 사항 (비전)

1.  **전체 맥락 분석**: 사용자의 *전체 글*을 RAG 지식 베이스 전체와 비교 분석
2.  **구조적 갭 분석**: 글이 RAG 데이터의 *어느 파트*에 해당하는지 파악하고, 해당 맥락에서 _무엇이 빠졌는지_ 분석
    - _예시_: "이 글은 RAG 데이터의 'Introduction' 챕터로 보입니다. 하지만 표준 Introduction에는 'Motivation' 파트가 포함되어야 하는데, 현재 글에는 없습니다."
3.  **매핑 입력**: RAG 데이터와 쉽게 매핑할 수 있도록 돕는 새로운 입력 필드 (메타데이터/태깅)

### 🗣️ 전문가 라운드 테이블

#### 👨‍💻 기술 리더 (기술적 타당성)

- **결론**: **가능합니다. 단, 비용이 듭니다.**
- **분석**:
  - 현재는 "Chunk 기반 검색" (작은 조각 검색) 사용 중. _전체 글_ vs _전체 RAG_ 비교는 연산량이 많음.
  - **해결책**: **"구조 매칭(Structure Matching)" 알고리즘** 필요. 단순 키워드 매칭이 아닌, 사용자 글의 *목차*와 RAG 문서의 *목차*를 비교.
  - **리스크**: RAG 데이터가 구조화되어 있지 않으면 어려움. 헤더/섹션이 있으면 충분히 구현 가능.

#### 📊 데이터 전략가 (비즈니스 가치)

- **결론**: **게임 체인저입니다.**
- **분석**: 대부분의 글쓰기 도구는 *문법(Form)*을 검사합니다. 디렉터님의 제안은 *알맹이(Content)*를 검증하는 것입니다.
- **가치 제안**: "콘텐츠 완성도 체크". 이것은 해당 주제를 아는 실제 인간 편집자를 흉내 내는 것입니다.
- **지표**: **"신뢰" 지표** 급상승 예상. 사용자는 AI가 언어뿐 아니라 주제를 진정으로 "이해한다"고 느낄 것입니다.

#### 🔎 프로덕트 리서처 (벤치마킹)

- **레퍼런스**: **Acrolinx** (콘텐츠 거버넌스 툴)
- **인사이트**: 엔터프라이즈 도구는 "콘텐츠 목표"를 사용합니다. "나는 *사용자 매뉴얼*을 쓰고 있다"라고 정의하면, "안전 경고"가 포함되었는지 확인해 줍니다.
- **차별화**: "매핑 입력" (주제/유형 선택기)은 고급 엔터프라이즈 도구의 **콘텐츠 프로필**에 해당합니다. 이는 우리를 "맞춤법 검사기"에서 **"콘텐츠 저작 플랫폼"**으로 격상시킵니다.

#### 🎨 UX/UI 전문가 (사용자 경험)

- **과제**: "빠진 부분"을 에러 로그처럼 보이지 않게 표시하는 방법?
- **아이디어**: "**청사진 오버레이(Blueprint Overlay)**". 왼쪽에 사용자 글, 오른쪽에 "이상적인 구조" (RAG 기반)를 체크리스트 또는 고스트 오버레이로 표시.
- **입력 필드**: "매핑 입력"은 단순한 텍스트 박스가 아닌, 스마트 **"컨텍스트 선택기"** (예: "무엇을 작성 중인가요? [보고서] [이메일] [논문]")

#### 🐣 주니어 개발자 (구현 재미)

- **아이디어**: "퍼센트 매치를 보여주면 어떨까요! '당신의 글은 Intro 섹션의 40%를 커버합니다'. 사용자에게 100%를 향한 명확한 목표를 줄 수 있어요."

### 🏁 합의 결과

- **업그레이드 vs 다운그레이드?**: **확실한 업그레이드.** 서비스를 "교정"에서 "가이드"로 진화시킵니다.
- **핵심 과제**: 성능 (분석 시간) 및 RAG 구조 명확화.

## 4. PRISM Writer 서비스 정체성 분석

### ❓ 디렉터 질문

"PRISM Writer는 단순한 검색 엔진인가요? 다른 RAG 서비스와 무엇이 다른가요?"

### 💡 기술 리더의 심층 분석

**"정정합니다. PRISM Writer는 단순한 검색 엔진이 아닙니다."**

PRISM Writer는 **AI 기반 콘텐츠 거버넌스 플랫폼**입니다.
일반 RAG 서비스가 *검색(Retrieval)*에 집중하는 반면, PRISM Writer는 **교정(Calibration)** (콘텐츠를 기준에 맞추는 것)에 집중합니다.

#### 🏗️ 핵심 차별화 요소

| 기능         | 일반 RAG           | PRISM Writer                         |
| :----------- | :----------------- | :----------------------------------- |
| **주 목적**  | 질문에 답하기      | **문서 개선하기**                    |
| **입력**     | 짧은 쿼리 (키워드) | **전체 초안 (컨텍스트)**             |
| **출력**     | 생성된 텍스트      | **평가 및 실행 가능한 패치**         |
| **인터랙션** | 채팅 인터페이스    | **듀얼 패널 워크샵 (에디터 + 코치)** |

#### 🧩 이미 구현된 핵심 컴포넌트

1.  **Rule Mining (`ruleMiner.ts`)**: 단순히 텍스트를 검색하는 것이 아니라, 지식 베이스에서 _암묵적 규칙_ (어조, 구조)을 추출합니다.

    - _차별화_: 일반 RAG는 사실을 찾습니다. 우리는 **패턴**을 찾습니다.

2.  **Template Gates (`templateGates.ts`)**: "가드레일"이 있습니다. RAG 데이터에 대한 **일관성** 및 **환각(Hallucination)** 여부를 검증합니다.

    - _차별화_: 일반 RAG는 콘텐츠를 생성합니다. 우리는 콘텐츠를 **검증**합니다.

3.  **Upgrade Plan (`evaluation.ts`)**: 단순 통과/실패가 아닌, 개선을 위한 로드맵을 제공합니다.

4.  **Synthetic Data Generator (`syntheticDataGenerator.ts`)**: 디렉터님이 언급하신 "500개 생성" 기능입니다. 이것은 우리의 **자기 개선 엔진(Self-Improvement Engine)**입니다. 자체 내부 모델을 파인 튜닝하기 위해 수천 개의 "골든 예시" (Q&A 쌍)를 생성합니다.
    - _차별화_: 일반 RAG는 정적입니다. PRISM Writer는 **스스로 학습하고 진화**합니다.

### 🚀 최종 결론

> **"PRISM Writer는 스스로 진화하는 AI 기반 콘텐츠 거버넌스 플랫폼입니다."**

디렉터님의 "구조 매칭" 요청은 우리 서비스 정체성의 **자연스러운 진화**입니다.

- 지금까지: **"마이크로 거버넌스"** (문장 단위 검사)
- 다음 단계: **"매크로 거버넌스"** (문서 전체 구조 검사)
